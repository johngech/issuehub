## Decision: Bun as Primary Runtime and Package Manager

**Date**: 2026-05-15
**Status**: Decided
**Owner**: Project owner

### Context

The project needed a runtime and package manager for both the server (Express) and client (React/Vite). The monorepo has `server/` and `client/` packages without a workspace config.

### Decision

Use **Bun** as the primary runtime and package manager for the entire project. Both packages run via `bun` commands. Bun auto-discovers monorepo packages via `bun.lock` without workspace configuration.

### Rationale

- Single runtime for both server and client development
- Built-in TypeScript support (no `ts-node` or `tsx` needed)
- Significantly faster package installs and script execution than Node.js
- Auto-discovers monorepo packages without workspace config files

### Alternatives Considered

| Alternative | Pros | Cons | Why Rejected? |
|-------------|------|------|---------------|
| Node.js + npm | Wider ecosystem maturity, more community packages | Slower installs, needs `tsx` or `ts-node` for TS | Bun provides better DX for a learning project |
| pnpm workspaces | Excellent monorepo support, strict | Requires workspace config, more overhead | Bun's auto-detection is simpler for this scale |

### Impact

- **Positive**: Simplified toolchain, fast installs, built-in TS support
- **Negative**: Some npm-only packages may have compatibility issues (unlikely for this stack)
- **Risk**: Bun ecosystem evolves fast — lock version for stability

## Decision: TanStack Router with File-Based Routing

**Date**: 2026-05-15
**Status**: Decided
**Owner**: Project owner

### Context

The React frontend needed a routing solution. File-based routing reduces boilerplate and enforces convention.

### Decision

Use **TanStack Router** with the `@tanstack/router-plugin/vite` plugin for file-based routing. The `routeTree.gen.ts` is auto-generated by the plugin — never edit manually.

### Rationale

- Convention-over-configuration: route files map to URLs automatically
- Auto code-splitting via `autoCodeSplitting: true` plugin option
- Type-safe route definitions and navigation
- Modern React patterns (loaders, actions, pending states)

### Alternatives Considered

| Alternative | Pros | Cons | Why Rejected? |
|-------------|------|------|---------------|
| React Router | More widely used, simpler mental model | Manual route config, no auto code-splitting | Less learning value for modern patterns |
| Next.js | Full framework, SSR, file-based routing | Heavier framework, more opinionated | Overkill for a learning SPA project |

### Impact

- **Positive**: Clean route structure, type safety, auto code-splitting
- **Negative**: Route tree auto-generation means a generated file in source
- **Risk**: Must remember NOT to edit `routeTree.gen.ts`

### Related

- `client/vite.config.ts` — Plugin configuration
- `client/src/router.tsx` — Router factory

## Decision: Tailwind CSS v4 with Vite Plugin

**Date**: 2026-05-15
**Status**: Decided
**Owner**: Project owner

### Context

The frontend needed a styling approach. Tailwind CSS v4 was the latest version with a new CSS-first configuration model.

### Decision

Use **Tailwind CSS v4** via `@tailwindcss/vite` plugin. Configuration is CSS-first (no `tailwind.config.*` file needed), integrated directly into Vite's build pipeline.

### Rationale

- Zero-config setup via `@tailwindcss/vite` plugin
- CSS-first configuration (v4 uses `@theme` directives instead of JS config)
- Utility-first approach enables rapid UI iteration
- Vite plugin means no PostCSS config needed

### Alternatives Considered

| Alternative | Pros | Cons | Why Rejected? |
|-------------|------|------|---------------|
| Traditional CSS files | Simple, no dependencies | More manual work, harder to maintain | Slower development speed |
| CSS Modules | Scoped styles, no class name conflicts | Verbose for rapid prototyping | Tailwind is faster for this scale |

### Impact

- **Positive**: Fast UI development, minimal config, modern approach
- **Negative**: HTML can look verbose with utility classes (acceptable trade-off)
- **Risk**: Tailwind v4 is relatively new — API stability

### Related

- `client/vite.config.ts` — Tailwind plugin registration

## Decision: Biome for Linting and Formatting

**Date**: 2026-05-15
**Status**: Decided
**Owner**: Project owner

### Context

The project needed code quality tooling. Traditional setups use ESLint + Prettier with many plugins and config files.

### Decision

Use **Biome** for both linting and formatting in both packages. Both `server/` and `client/` have independent `biome.json` configs, both using tab indentation and double quotes.

### Rationale

- All-in-one tool replacing ESLint + Prettier
- Significantly faster performance (written in Rust)
- Built-in import organization (`organizeImports`)
- Lets Biome handle both packages independently

### Alternatives Considered

| Alternative | Pros | Cons | Why Rejected? |
|-------------|------|------|---------------|
| ESLint + Prettier | Massive plugin ecosystem, widely adopted | Slow, many config files, plugin conflicts | More overhead for a learning project |
| dprint | Fast, Rust-based | Smaller ecosystem, fewer rules | Biome has better community adoption |

### Impact

- **Positive**: Single tool, fast, minimal config, consistent formatting
- **Negative**: Fewer plugin options than ESLint ecosystem
- **Risk**: Biome 2.x is mature but some niche rules may be missing

### Related

- `server/biome.json` — Server Biome config
- `client/biome.json` — Client Biome config (`routeTree.gen.ts` and `styles.css` excluded)

## Decision: Vitest + Testing Library for Testing

**Date**: 2026-05-15
**Status**: Decided
**Owner**: Project owner

### Context

The project needed a testing framework. Only the client has tests configured at this stage.

### Decision

Use **Vitest** (v4) with **Testing Library** (`@testing-library/react` and `@testing-library/dom`) and **jsdom** for browser environment simulation.

### Rationale

- Native Vite integration — same transform pipeline as the app
- Fast (HMR for tests, parallel execution)
- Modern API compatible with Jest but faster
- jsdom provides DOM APIs for component testing without a browser

### Alternatives Considered

| Alternative | Pros | Cons | Why Rejected? |
|-------------|------|------|---------------|
| Jest | Mature, widely used | Slower, more config needed, no native Vite integration | Vitest is faster and integrates better |
| Playwright | Best for e2e testing | Heavier, overkill for unit/component tests | Can add later for e2e if needed |

### Impact

- **Positive**: Fast test execution, modern patterns, good DX
- **Negative**: Only client tests configured (server testing future)
- **Risk**: None significant — Vitest is stable

### Related

- `client/package.json` — Test dependencies

## Decision: OpenAgentsControl (OAC) for AI-Assisted Development

**Date**: 2026-05-15
**Status**: Decided
**Owner**: Project owner

### Context

The project wanted AI-assisted development with project-specific context awareness.

### Decision

Install and configure **OpenAgentsControl** in `.opencode/` providing OpenAgent, OpenCoder, and 7+ subagents (ContextScout, TaskManager, CoderAgent, TestEngineer, CodeReviewer, BuildAgent, ExternalScout).

### Rationale

- Project-specific AI context (unlike generic LLM chat)
- Custom commands and specialized subagents
- Project intelligence files maintain shared understanding

### Alternatives Considered

| Alternative | Pros | Cons | Why Rejected? |
|-------------|------|------|---------------|
| Cursor IDE | Good AI features, built-in | Proprietary, less customizable | OAC is open and project-embedded |
| Raw LLM chat | Universal, no setup | No project context, no tool integration | OAC provides structured context |

### Impact

- **Positive**: AI with full project context, structured documentation
- **Negative**: Additional framework to learn and maintain
- **Risk**: OAC ecosystem maturity

### Related

- `AGENTS.md` — AI agent instructions
- `.opencode/` — OAC framework directory
